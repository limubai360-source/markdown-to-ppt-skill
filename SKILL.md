# Markdown 转 PPT

## 总览

使用本技能把一个 Markdown 想法稿转换为 PPT。流程必须保留两个确认点：先让用户确认 Markdown 格式的 PPT 大纲，再让用户确认 HTML 样稿；只有用户明确要求时才继续生成 PPTX。

本技能面向 Codex、Claude Code、OpenCode、OpenClaw 等代码 Agent。不要依赖 Codex 专属工具、浏览器插件或 MCP。可以读取本技能目录下的脚本和参考文件，并按当前 Agent 可用的 Node.js 环境执行。

## 语言规则

- 自动识别输入语言：中文稿默认输出中文，英文稿默认输出英文，中英混合稿默认保留双语语境。
- 用户要求“双语 PPT”时，每页生成中英文对应内容。推荐结构是中文主标题 + 英文副标题；正文按信息密度选择并列双语、上下分层或重点词双语标注。
- 不要机械逐句翻译。优先保持演示叙事自然、页面可读、术语一致。
- 用户未要求双语时，不额外扩写另一种语言。

## 标准流程

### 1. 读取并理解 Markdown

读取用户提供的 Markdown 文件或文本。判断原稿结构：

- 如果章节、层级和论述顺序已经清楚，保留其主线并压缩成 PPT 叙事。
- 如果章节不清楚，重新理解并分章节整理。
- 章节层级默认 2 层，最多 3 层。不要生成超过 3 层的 PPT 大纲。
- 删除重复、跑题和弱相关内容，但保留原文中的关键观点、数据、例子、结论和限制条件。

### 2. 输出 PPT 大纲并等待确认

先输出 Markdown 格式的大纲，不要直接生成 HTML。大纲建议包含：

```markdown
# PPT 大纲 / Slide Outline

## 1. 封面
- 标题：
- 副标题：

## 2. 章节名
### 2.1 幻灯片标题
- 核心观点：
- 页面内容：
- 建议视觉：
```

大纲确认提示要简短明确，例如：“请确认这个大纲方向是否可行；如果没问题，我会继续生成 HTML 样稿。”

### 3. 选择样式

读取 `references/styles.md`。如果用户没有指定样式，默认使用样式一：Linear / Notion。四种样式为：

- 样式一：Linear / Notion，适合结构清晰、现代、高效、产品化。
- 样式二：Apple Keynote 极简，适合高级、愿景感、未来感、聚焦。
- 样式三：咨询公司风格，适合专业、可信、逻辑严密、适合决策。
- 样式四：现代 SaaS 官网风格，适合智能、开放、平台化、增长、产品价值。

### 4. 生成 HTML 样稿

根据确认后的大纲和所选样式生成完整单文件 HTML。HTML 样稿的目标不是“能翻页”，而是：**可演示、可审美检查、可转 PPTX 的静态幻灯片设计稿**。

#### HTML 硬性结构要求

- 页面比例为 16:9。
- 每页使用 `<section class="slide">`，封面使用 `<section class="slide cover">`。
- CSS 写在同一 HTML 文件内，必要的轻量 JS 也必须内联。
- 不依赖外部图片资源，除非用户明确提供或要求；优先使用 CSS 形状、标签、卡片、线框箭头、表格和流程结构。
- 不依赖复杂外部库；不要把 Slidev、Reveal、插件运行时目录、导航控件、缩略图栏等作为最终样稿的一部分，除非用户明确要求使用该框架。
- 所有内容必须能被后续脚本解析成 PPTX。
- 推荐输出文件名：`<topic-slug>-slides.html`。

#### Linear / Notion 默认审美范式

- 整体简约、清晰、轻量，强调信息层级和可读性。
- 背景以白色、浅灰、淡蓝灰为主，避免大面积深色背景。
- 配色以黑、白、灰为主，使用少量蓝色或紫色作为强调色；避免鲜红色成为主视觉。
- 使用卡片式布局，每个信息模块放在圆角卡片中，卡片只使用浅边框或轻微阴影，不要厚重。
- 字体接近 Inter / Helvetica / PingFang SC；标题清晰，正文克制。
- 页面气质偏产品文档感，适合展示问题、方案、能力模块、产品价值、决策逻辑。
- 图形元素优先使用简洁线框图标、浅色标签、模块卡片、轻量箭头、矩阵和对比结构。
- 每页信息密度适中：不要只有标题和一句观点；通常应包含一个明确观点、一个结构化组件和必要解释。
- 表格页要重排为卡片、矩阵、对比表或流程图，而不是直接堆文本。

#### 自适应和版式要求

- 幻灯片必须在 2K / 4K 屏幕、普通笔记本和浏览器窗口中都保持 16:9 自适应：不能出现画布变大但内容很小、留白过大、内容被裁切的问题。
- 推荐使用如下思路控制画布：

```css
.deck {
  width: min(100vw, calc(100vh * 16 / 9));
  height: min(100vh, calc(100vw * 9 / 16));
  container-type: size;
}
```

- 内容尺寸优先使用容器单位（如 `cqw` / `cqh`）或基于 slide 的 CSS 变量，不要只依赖全局 `vw` / `vmin`。
- 标题、正文、卡片、表格、流程图都应随 slide 容器缩放；内页标题通常应至少是正文的 1.8 倍。
- 普通内容页的主体区域应使用约 55%-75% 的 slide 高度；不要把两张小卡片放在上半屏后留下大片空白。
- 如果某张卡片内容较少，应通过更强的层级、补充解释、标签、结论栏或结构图来填满表达，而不是让卡片成为小条。
- 表格页默认应完整呈现关键内容，不依赖点击后才能看懂。

#### 动画要求

- 动画用于增强讲述节奏，不应隐藏理解一页所需的关键内容。
- 默认推荐：进入页面后，当前页模块按顺序自动播放进入动画；键盘下一步只负责翻页。
- 如果用户要求逐步播放，则应做到“一次按键只出现一个模块”，并在动画期间锁定输入，避免一次点击触发多个动画或直接跳页。
- 不要同时给鼠标和键盘绑定互相叠加的翻页/动画逻辑，避免鼠标一下快速播完所有动画和页面。
- 动画时长以 500-800ms 为宜，使用轻微位移（8-18px）、透明度和柔和缓动；避免复杂、花哨、难转 PPTX 的动画。

HTML 必须同时包含 `pptx-spec`：

```html
<script type="application/json" id="pptx-spec">
{
  "version": 1,
  "size": "LAYOUT_WIDE",
  "theme": "linear-notion",
  "slides": [
    {
      "type": "cover",
      "title": "标题",
      "subtitle": "副标题",
      "background": "#F8FAFC",
      "elements": [
        {
          "kind": "text",
          "text": "标题",
          "x": 0.8,
          "y": 1.0,
          "w": 11.7,
          "h": 0.8,
          "fontSize": 34,
          "color": "#111827",
          "bold": true
        }
      ]
    }
  ]
}
</script>
```

`pptx-spec.slides` 数量必须与 `section.slide` 数量一致。坐标单位使用英寸，16:9 宽屏为 13.333 x 7.5。

支持的 `elements.kind`：

- `text`：文本块，支持 `text`、`x`、`y`、`w`、`h`、`fontSize`、`color`、`bold`、`align`、`valign`。
- `card`：圆角卡片，支持 `x`、`y`、`w`、`h`、`fill`、`line`、`radius`、`title`、`body`。
- `tag`：轻量标签，支持 `text`、`x`、`y`、`w`、`h`、`fill`、`color`。
- `shape`：基础图形，支持 `shape`、`x`、`y`、`w`、`h`、`fill`、`line`。
- `line`：线条或箭头，支持 `x`、`y`、`w`、`h`、`color`、`arrow`。

### 5. 自我矫正

在交付 HTML 前必须主动检查并修正，不要把问题留给用户指出。

#### 内容和叙事检查

- 是否每页都有一个清晰观点，而不是只有主题词？
- 是否存在“标题页过多、正文过空”？
- 是否把 Markdown 的线性层级误当成 PPT 页面结构？
- 是否把长段落转成了可演示结构：结论、因果、对比、流程、矩阵、清单、案例？
- 方法论页是否说明“解决什么问题、怎么用、容易错在哪里”？
- 案例页是否有情境、动作、结果、可复用经验，而不是只保留案例标题？
- 图表形式是否与内容匹配，每页是否过满或过空？

#### 视觉和版式检查

- 是否遵守用户指定风格；若未指定，是否符合 Linear / Notion 默认审美？
- 是否避免了大面积深色、鲜红主色、厚重装饰和过度营销感？
- 字体是否过小，尤其是在 2K / 4K 屏幕预览时是否出现“画布很大、内容很小”？
- 内容区是否占据足够空间，是否存在大片无意义留白？
- 标题、卡片、表格、流程图是否对齐一致，层级清晰？
- 表格页是否美观：表头、分组、斑马纹/浅底、重点标签、足够行高、无拥挤或过空？
- 背景与字体颜色是否过近导致难以辨识？
- 是否所有页都适合演示，而不是像文档截图？

#### 交互和动画检查

- 如果有动画：是否能清晰看见，而不是一闪而过？
- 若设置自动播放：是否进入页面后自动完成当前页动画，键盘只翻页？
- 若设置按键分步：是否一次按键只推进一个动画步骤？
- 是否避免鼠标和键盘事件重复触发导致快速播完所有动画和页面？
- 是否没有目录栏、侧边栏、调试控件、框架导航条等运行时 UI 干扰最终样稿？

#### 浏览器抽检

- 对封面、章节页、双栏页、卡片页、表格页、流程页、结尾页各抽查至少一页。
- 在当前浏览器尺寸下检查 16:9、自适应、字体大小、留白、动画、表格美观度。
- 修改用户反馈时，要保留前面已经修好的问题：例如目录栏、蓝白配色、避免红色、动画节奏、自适应、表格排版等，不要修一处丢一处。

修正后运行：

```bash
node /path/to/markdown-to-ppt/scripts/validate_deck_html.js <slides.html>
```

### 6. 交付 HTML 并等待确认

把 HTML 文件路径交给用户，并说明已经完成自检。询问是否有修改点。若用户提出修改，更新 HTML 后再次执行第 5 步。

### 7. 可选生成 PPTX

只有用户明确要求“生成 PPTX”或“根据 HTML 样稿生成 PPTX”时才执行：

```bash
node /path/to/markdown-to-ppt/scripts/html_to_pptx.js <slides.html> <slides.pptx>
```

PPTX 要求：

- 每个 `section.slide` 对应一页幻灯片。
- 比例为 16:9。
- 尽量复刻 HTML 的视觉风格、布局、字号、颜色、卡片和图形关系。
- PPTX 转换追求高保真近似，不承诺逐像素还原浏览器渲染。
- 牢记：HTML/CSS 视觉稿不会自动等于 PPTX；PPTX 的视觉质量取决于转换工具或 `pptx-spec` 是否同步了 HTML 的字体、颜色、卡片、表格和版式。
- 如果先生成 HTML，再生成 PPTX，必须让 `pptx-spec` 对齐 HTML 的：
  - 页面数量和章节结构。
  - 字体族、字号层级、行高。
  - 主色、强调色、背景色、边框和阴影风格。
  - 卡片布局、表格样式、标签、箭头、流程线。
  - 封面和结尾页配色。
- 如果用户提供标准 PPT 模板，要先判断模板的作用：
  - 它可能只是品牌规范，不等于美观范式。
  - 不要机械套模板导致内容空、线框乱、风格僵硬。
  - 必要时用模板约束品牌元素，用 HTML 样稿或参考设计约束审美与版式。
- 转换后必须至少做基本检查：
  - 文件是否存在。
  - 页数是否合理。
  - 是否明显丢失中文、表格、布局或配色。
  - 是否出现与 HTML 不一致的配色、字体、角标题、卡片、表格和间距。

如果缺少 `pptxgenjs`，脚本会提示安装方式。Codex 通常可使用 bundled Node runtime；Claude Code、OpenCode、OpenClaw 可在本地项目安装 `pptxgenjs` 后运行。

## 脚本

- `scripts/validate_deck_html.js`：检查 HTML 是否符合可转 PPTX 的基本结构。
- `scripts/html_to_pptx.js`：根据 HTML 中的 `pptx-spec` 生成 PPTX。

## 输出纪律

- 先大纲，后 HTML，再可选 PPTX。
- 不要跳过用户确认点。
- 不要把样式说明塞进幻灯片正文。
- 不要输出依赖外部图片或在线 CSS 的 HTML。
- 不要生成超过 3 层的章节结构。
- 不要在用户未要求时自动生成 PPTX。
- 不要把“能生成”当成“做得好”；必须主动检查审美、信息密度、自适应、动画和可转换性。
- 当用户针对上一版提出反馈时，要把反馈当成累计约束：后续版本必须保留已解决的问题，不要反复退回到目录栏、配色不一致、字体过小、留白过大、动画失控、表格粗糙等旧问题。
- 如果某个框架（如 Slidev）带来目录栏、点击状态、运行时控件、PPTX 不可控等问题，应优先回到单文件 HTML 样稿或静态导出，而不是继续修补框架控制层。
